# `Angular`开发小程序

- 通过修改`Angular`实现使用`Angular`开发小程序
- 本项目实现的目的很简单也很纯粹,就是告诉大家.框架无高低,使用的人,有高低

## 修改内容

- builder
  > 修改用于支持自定义构建
- component 编译时
  > 通过提前解析获得模板与样式,转换为小程序所需的
- 全局变量
  > 用来支持`ngZone`与`Angular`默认全局引用
- platform
  > 通过实现自定义的 platform,给予小程序的启动上下文支持
- renderFactory
  > 原渲染工厂支持的是 dom 渲染,但是小程序不需要
- 主出口重建
  > 主出口预先加载相关依赖,为后面的 page 做准备
- 文件系统监听修改
  > 需要在预处理结束时,清除文件缓存

## 增加内容

- 组件与页面的注册函数
  > 只有被注册,还会自动的查询依赖关系与生成
  > 组件目前改为仅注册,生成由 ng 默认逻辑自动实现
- 相关生命周期的传递
  > 小程序自带的一些生命周期
- 小程序相关 token.
  > `APP_TOKEN`可以获得 App 实例
  > `PAGE_TOKEN`可以获得组件对应的小程序页面实例

## 已实现内容

- 变更检测
- Input,Output
- 依赖注入
- ng-content
- 小程序大部分功能
- 管道
- 属性型指令
  > 属性型指令目前只支持源码开发,也就是你自己写的,不支持第三方库...比如`@angular/forms`
- 自定义表单组件
  > 从逻辑上是支持的,但是我没有测试......
  > 表单指令和自定义表单组件看起来像一个东西,实际上是两个
- library
  > 组件与指令的编写与 web 基本一致
- 组件 style
  > 由于升级到 ng13 某些内部包不能被直接使用,所以现在仅支持 styleUrls 里面的样式进行编译,不要使用内联的 styles
  > 未来有空可能会找其他方法实现,但是由于这个不是太严重,所以优先级不高
- 表单
  > input 已实现
  > switch 已实现
  > slider 已实现
  > radio 已实现
  > checkbox 已实现
  > picker 已实现
  > 剩下的将由第三方实现
- 有限结构型指令
  > 所谓有限就是在当前组件下`<ng-template></ng-template>`实例可以在任意地方引用
  > 自定义的结构型指令可以任意引用,但是传入创建内嵌模板时,需要一个 name,name 与#xxx 应一致
  > common 下的由于是先实现的,所以不支持任意应用只支持静态的使用
  > common 可以通过与 forms 一样的 Hook 方式实现任意引用,但是需要调研各种不同平台的小程序语法是否几乎完全相同,如果没有完全相同,差别过大,可能需要其他方式实现
  > ngIf,ngFor,ngSwitch,ngTemplateOutlet

# 无法实现

- ssr 不可能...如果能实现 ssr 我也不会费那么大劲来适配小程序
- pwa 两者处于竞争对手....
- 无限制的结构型指令
  > 除非模板运行动态按需加载
  > 理论上可以把所有模板都变成全局模板,然后所有组件全部 import 这个全局模板,每个模板都有独一无二的 id,但是这样即使能实现也有一个问题,就是代码都糊一块的.使用一个就等于使用使用所有代码(毕竟一个结构型指令就会生成一个模板)
  > 即使这个功能实现了,也可能会导致实现的滥用而导致代码效率变低,所以...对这个功能比较谨慎
  > 如果能证明模板被引入但是不被使用的前提下不会影响解析速度和加载速度点话,可以实现

# 未实现不影响整体功能以后可能实现

- i18n 众所周知,国内使用 i18n 的并不多,使用 ng 内置的 i18n 功能就更少了,目前内置的 i18n 并不支持,但是可以用第三方支持
- 单元测试 同理,目前来说单元测试比 i18n 重要些,但是由于平台的不开放性,所以如果支持需要自己模拟一个环境,但是这个也不影响开发
- 动画 这部分没有仔细研究,但是也有 css 可以稍微代替下,优先级也不高
- 路由 没有多级路由,实现起来比较鸡肋
